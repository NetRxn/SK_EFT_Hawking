## Context

### The mandate, and the gap

The repository's standing purpose is to map the proof landscape on **both fronts** — what is proved (positive) and what is refuted (no-go / obstruction) — across **many concurrent phases and waves** working separate physics areas (Phase families 1–7 with deep alphabetic sub-series; ~92 roadmaps advancing independently). The declaration dependency graph is already captured programmatically and already drives paper provenance (CHECK 14) and QA (the claims-reviewer). What is **missing** is a single *view* that, per result, answers: **positive / negative / open** — and, for the open set, **what is provable now** (every proof-dependency already closed). That view is what we call the **atlas**.

### The substrate already exists — and it is derivation-based

The survey's load-bearing finding is that almost everything an atlas needs is already computed:

- **`ExtractDeps.lean` → `lean_deps.json`** walks the *whole environment by package membership* (`inSKPackage`, module-path prefix `SKEFTHawking`), not a maintained import list. It emits, per declaration: `name` (FQN), `kind` (theorem/def/axiom/structure/inductive/instance), `module`, pretty-printed `type`, the **transitive axiom closure split** `axiom_deps_project` vs `axiom_deps_core`, `structure_fields`, and — opt-in via `EXTRACT_NAME_DEPS=1` — `name_deps_project`, the **declaration-level proof-dependency edges** (who-invokes-whom). Re-extraction is SHA-triggered and automatic; **a new wave's module + one `import` line in `SKEFTHawking.lean` makes all its decls appear with zero registry edit.**
- **`build_graph.py` → AGE graph `sk_eft`** lifts that into a typed, **FQN-keyed** (`lean:<full_name>`) property graph — 22 node labels, ~28 edge types — whose AGE labels are created *type-agnostically at write time*, so a new node/edge type needs **no schema change**. It already carries the proof-dep layer (`USES`, gated `SK_EFT_INCLUDE_USES=1`) and the trust-boundary layer (`DEPENDS_ON_AXIOM`). Crucially it is **JSON-first**: the primary output is an in-memory `{nodes, links, meta}` view; the Postgres/AGE write is strictly **opt-in** (`--sync-pg`), and every dev-time consumer (`graph_integrity`, `readiness_gates`, the dashboard cache) reads the JSON, never the DB.
- The **non-positive states already have homes**: no-go theorems are proved negations in dedicated modules by the `_no_go | _nogo | _refuted | _obstruction | _falsified` convention (a regex `validate.py` already recognizes); open assumptions are FQN-keyed in `HYPOTHESIS_REGISTRY` (~33 tracked Props, each with `dependent_theorems` = outgoing edges) and `AXIOM_METADATA` (0 live project axioms today); open *targets* are `tier='discharge_future'` / `status='proposed'` Props.

### The temptation, and why we reject it

The obvious move — a separate hand-maintained atlas (a node registry plus one-Lean-file-per-node) — would (a) **duplicate** a graph we already build, (b) require **per-wave hand-registration**, which directly violates the hard requirement that *arbitrary later phases and waves fold in automatically without disrupting anything else*, and (c) **drift** from source. The project already keeps **100% of metadata out of theorem files** (external Python registries + an env-walking extractor that reads no docstrings/attributes; there are **zero custom Lean attributes** in the tree today). An atlas must preserve that separation — both to honor the no-drift discipline and to keep proof code **upstreamable to Mathlib/Physlib** (Physlib is a pinned dependency on our exact Mathlib rev; the `*MathlibPR.lean` modules are the in-repo upstream template).

## Decision

Adopt a **Derived Proof Atlas**: a classification + frontier-ranking + integrity layer **projected onto the existing dependency graph**, materialized **JSON-first** (so dev, the swarm scheduler, and CI never require a running database) and synced into the `sk_eft` AGE graph for Cypher when Postgres is up. The atlas is a *view*, not a store; its source of truth is `lean_deps.json` + the existing registries, never a parallel node file.

### Upstream cleanliness (a first-class constraint, not an afterthought)

All atlas metadata stays **out of theorem statements and proofs**. The only in-source addition is the D-E `@[atlas_node]` attribute, which is non-load-bearing and strippable; everything else is derived or lives in external registries. Upstream-bound code (the `*MathlibPR.lean` modules and other clean helper lemmas) is **auto-`TRUE`** and carries **no** attribute. The rare declaration that is both an atlas apex *and* Mathlib/Physlib-bound drops its attribute line on PR — mechanical, and rare in practice because headline/apex theorems are mostly "keep-local" per `UpstreamContributionDisposition.md`. The kernel-purity bar an atlas `TRUE` node must meet is **identical** to the project's existing standard (closure ⊆ `{propext, Classical.choice, Quot.sound}`, zero sorry, no `native_decide` regression, no `maxHeartbeats` in proof bodies), so atlas nodes are upstream-clean by construction. The extractor that reads the attribute is project tooling (like `ExtractDeps`), not shipped math.

### Rollout (each phase independently shippable behind the wave gate)

1. **Phase 1a — derived atlas, zero attributes.** Wire D-G (emit `proof_deps`/`statement_deps` as an `AxiomClosure` byproduct; retire the redundant walk) and D-G.1 (reuse `regen_lock` at the extraction chokepoint + lead-only); add the `atlas_view` classifier (D-B/D-C/D-D) over `lean_deps.json` ∪ `HYPOTHESIS_REGISTRY` ∪ `AXIOM_METADATA` ∪ the no-go scan; emit `meta.atlas_*` + `IMPLIES` into the graph; ship `validate.py --check atlas_integrity` (D-F). The only Lean-side change is the `AxiomClosure` return-value refactor (infrastructure, not proofs); **no theorem `.lean` change required**. **Bench gate:** confirm fast-path wall-time + `lean_deps.json` size at 23k decls before proceeding.
2. **Phase 1b — reduce the extraction cost (D-G.2, IMPLEMENTED 2026-06-22).** Three pin/`declHash`-keyed caches: the **Mathlib boundary axiom-closure leaf cache** (the precompute — closure Tarjan **263 s → 24 s**, ~10×), the per-decl `typeStr`/`ppExpr` cache, and the per-decl immref cache. Same-host cold↔warm output is **byte-identical**; extraction total **514 s → 262 s (~2× steady-state)**. The remaining warm floor (env-scan ~91 s + the `processDecl` loop ~134 s) was the **Phase 1c** target: **IMPLEMENTED 2026-06-22** as a per-decl **JSON-string cache** keyed on `(declHash, kind, module, hash(axioms))`, with the render logic drift-proofed by hashing `ExtractDeps.lean`'s source into the pin (zero hand-maintained key). The loop's cost measured as **124 s of JSON build** (`toString`-of-`Name`), not `classifyDecl`. On warm hits the whole build is skipped → **warm 262 s → 134 s**; output switched to compact JSON (data-identical). **Net cold 514 s / warm 134 s (~3.8× steady-state).** Phase 1d (parallelism / env-scan) is possible but lower-ROI — extraction runs at wave boundaries, not per-edit.
2. **Phase 2 — attribute curation.** Add the `@[atlas_node]` attribute (D-E) + the `ExtractDeps` reader; annotate landmarks (apex/track/no-go-where-naming-fails) incrementally; promote phase/wave to derivable (D-H).
3. **Phase 3 — surfaces + swarm.** A heatmap/dashboard view over the derived frontier; wire the swarm scheduler's frontier consumption (D-I).

## Consequences

**Positive.** A single, queryable, **auto-extending** map of the proof landscape on both fronts (positive / negative / open) plus a live *provable frontier*, derived from the substrate already maintained — no parallel store, no per-wave hand-registration, no theorem-file pollution. The atlas becomes first-class graph data (Cypher-queryable alongside papers/QA when PG is up) yet remains fully available DB-free. The honesty firewall (apex-never-silently-closed, AXIOM_TAINTED) is the same soundness contract the 0-axiom/0-sorry posture already enforces, now made explicit for open targets.

**Costs / risks.** (1) D-G is **structurally free** at runtime (the `getUsedConstantsAsSet` traversal is already paid by `AxiomClosure` on the fast path; we reuse its `depCache` and retire the redundant `EXTRACT_NAME_DEPS` walk) — the residual costs are a larger `lean_deps.json` (more edges) and a heavier graph edge-assembly, both to be benched once at 23k decls, plus the small implementation lift (the `AxiomClosure` refactor to return per-decl project-local refs + the D-G.1 extraction lock). The earlier "~5–10 min slow path / 250 ms-budget dropped-edge" concern is **eliminated** by the byproduct approach, not merely tolerated. (2) D-E introduces the project's **first custom Lean attribute** — a one-time infra cost (a small `AtlasAttr` module + an `ExtractDeps` reader) and a check that it does not trip Mathlib linters on any upstream-bound file (mitigated by keeping it a namespaced, non-load-bearing label; the fallback if it proves troublesome is an external FQN-keyed registry like `HYPOTHESIS_REGISTRY`, at the cost of drift). (3) The `TRUE`-vs-`OBSTRUCTION` classifier has false-positive risk (a no-go is kernel-identical to any theorem) — mitigated by the naming convention + D-E one-bit hint + the `atlas_integrity` gate's no-double-kind invariant.

**Explicitly out of scope.** Re-deriving any genuinely Mathlib-walled mathematics (`topo`/Rokhlin, prime-density, the named literature walls). The atlas **classifies and surfaces** the proof state of those nodes (as `UNKNOWN`/tracked-hypothesis, exactly the ADR-003 posture); it never demands they be discharged. The D-E attribute is curation, never proof.

## Alternatives considered

1. **Separate hand-maintained atlas registry (file-per-node + a node list).** Rejected — duplicates the graph, requires per-wave hand-registration (violates the auto-fold-in requirement), and drifts from source. (File-per-node is, separately, a *swarm-parallelism* device for conflict-free merges, not a data-model necessity; a derived atlas keys by FQN and tolerates many nodes per file.)
2. **Atlas only inside the AGE/Postgres graph.** Rejected — would couple everyday development and the swarm scheduler to a running database, which is not always up during dev. D-A's JSON-first form gives the graph-projection benefits without that hard dependency.
3. **Infer everything from naming/namespace, no attribute at all.** Rejected as the *sole* mechanism — brittle for nodes that don't fit the naming convention, and unable to express `is_apex`, `track`, or curated cross-edges. Naming is the default; the D-E attribute is the minimal, strippable escape hatch for the irreducible residue.
