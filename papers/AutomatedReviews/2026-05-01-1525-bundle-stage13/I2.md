## Round-2 finding remediation verification

### 7.1 (Gate 9, §5 number-field count drift) — REMEDIATED

- **Round-2 issue:** §5 stated per-module counts QCyc3=11, QCyc15=18,
  QCyc15SqrtPhi=10 with cumulative "approximately 161" hedged via
  "convention-sensitive recounts ... yield ~+3 above this baseline".
  Round-2 review showed the +3 was actual drift (not convention
  sensitivity) and the alt convention adds +43, not +3.
- **Round-3 state:** Paper §5 lines 500-502 now read:
  ```
  \texttt{QCyc3.lean} (12), \texttt{QCyc5.lean} (29),
  \texttt{QCyc5Ext.lean} (36), \texttt{QCyc15.lean} (19),
  \texttt{QCyc15SqrtPhi.lean} (11), and \texttt{QCyc16.lean} (12)
  ```
  Paper §5 line 515 now reads:
  ```
  The cumulative count is 164 declarations across the nine
  algebraic-number-field modules.
  ```
  The hedging clause has been dropped.
- **Verification:** Direct line-start grep on each of the nine
  number-field modules using the convention `theorem|lemma|def|noncomputable def|structure|class|inductive|abbrev`:
  ```
  $ for f in QSqrt2 QSqrt3 QSqrt5 QCyc3 QCyc5 QCyc5Ext \
            QCyc15 QCyc15SqrtPhi QCyc16; do
      grep -cE '^(theorem|lemma|def|noncomputable def|structure|class|inductive|abbrev) ' \
        lean/SKEFTHawking/${f}.lean
    done
  → 11 20 14 12 29 36 19 11 12   (total 164)
  ```
  Per-module: QSqrt2=11, QSqrt3=20, QSqrt5=14, **QCyc3=12**,
  QCyc5=29, QCyc5Ext=36, **QCyc15=19**, **QCyc15SqrtPhi=11**,
  QCyc16=12. Sum **164**, exactly as paper claims. The three
  changed numbers (bolded) match.

  Note: under the strict "no-`abbrev`" convention (omit `abbrev`
  from the keyword set), the totals would be 11/20/14/11/29/36/18/10/12 = 161;
  but `abbrev QCyc3 := PolyQuotQ 2`, `abbrev QCyc15 := PolyQuotQ 8`,
  and `abbrev QCyc15SqrtPhi := PolyQuotOver QCyc15 2` are top-level
  user declarations (not internal `private`/`protected` shorthand)
  and counting them as `def`s under `defnInfo` is the convention
  Lean's own `lean_deps.json` extractor uses (ExtractDeps.classifyDecl
  maps `defnInfo` → `"def"`, which catches all `abbrev`s). So 164
  is the principled number; 161 was the under-count from naive
  shell-grep that excluded `abbrev`.
- **Status:** **REMEDIATED**, gate cleared.

Optional clarity nit (NOT a finding, NOT advisory):
the paper does not state the convention upstream of §5 line 515
either — a reader doing their own `grep -E '^(theorem|lemma|def|...) '`
without `abbrev` will find 161 and may briefly wonder where the +3
comes from. A single-clause addition like "(line-start counts of
\texttt{theorem}, \texttt{def}/\texttt{abbrev}, \texttt{structure},
\texttt{class}, \texttt{inductive} declarations)" upstream of
"164 declarations" would be a quality-of-life improvement. This
is a *style*, not a *correctness*, observation; the number itself
is reproducible and correct, so no finding emitted. Future-self
hint for paper polish, not a Stage-13 gate concern.
