## QI Candidate

**Pattern: post-bundle-lift bibitem drift introduces author-list errors
that pass cache-lookup tests because the registry mirrors the bibitem.**

- The L3 `Balbinot2005PRD` author-list error (finding 1.1) propagated
  from a (presumed) per-paper edit during the Phase 7b sub-wave 7b.1
  lift from paper27. Paper27 has the correct 3-author list. The
  registry's `Balbinot2005PRD` entry has the same 4-author error as L3,
  suggesting the registry was edited to *match* the lifted bibitem
  rather than the registry being the source-of-truth.
- This is a Gate 1 (CitationIntegrity) failure mode the existing pipeline
  doesn't catch because `validate.py --check citation_primary_sources_present`
  only verifies the cache file *exists*, not that the registry entry's
  author list agrees with the cache. The pdf is correct; the registry
  is wrong; the bibitem matches the registry; round-trip self-consistency
  hides the underlying ground-truth violation.
- **Proposed mitigation:** extend `validate.py` (or add CHECK 22+) to
  diff the registry's `authors` field against a parsed author list from
  the primary-source cache (`Balbinot2005PRD.source/acusticpap.tex:57-67`,
  `Balbinot2005PRD.json` arXiv metadata, etc.). The diff catches the
  current failure mode + is robust across future bundle lifts.
- **Cross-bundle reach:** the duplicate-`inprep`-keys regression (finding
  1.2) affects ALL `Roehm2026X` self-cites (D3, D4, D5 confirmed by grep
  in citations.py:7411–...), so the same mechanism that escapes Pipeline
  Invariant #11 in the L3 review applies to every bundle that cites a
  Roehm2026 in-prep companion. Closes via the
  `feedback_promote_primary_sources_duplicate_keys.md` memory action
  item (Wave 5+ infrastructure fix).
